UNDERSTANDING CU vs TTY DEVICES
================================================================================
macOS creates two device files for each serial port:

/dev/cu.* (Call-out device):
  - Use this for most applications
  - Non-blocking behavior
  - Preferred for serial communication
  - Does not wait for carrier detect

/dev/tty.* (Terminal device):
  - Traditional terminal device
  - Waits for carrier detect signal
  - Use only if you need terminal semantics

RECOMMENDATION: Use /dev/cu.* devices for mesh node communication

TROUBLESHOOTING
================================================================================

Issue: "Resource busy"
Solution: Only one process can access a serial port at a time.
         Close other applications (screen, cu, Chrome) before connecting.

Issue: "Permission denied"
Solution: Your account already has full access (666 permissions).
         This should not occur.

Issue: No data received
Check:
  - Correct baud rate (115200 is most common)
  - Proper ground connection
  - TX/RX not swapped
  - Device is powered on
  - Firmware is running

Issue: Garbled data
Check:
  - Baud rate mismatch
  - Wrong parity settings
  - Flow control issues
